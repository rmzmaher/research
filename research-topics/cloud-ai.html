Cloud-Native AI & MLOps for Upstream Analytics

Feature stores, model registries, CI/CD pipelines, and agentic AI architectures for scalable deployment of production and reservoir analytics in cloud environments.

1. Scope of the Research Area

Machine learning models for production surveillance, virtual flow metering, and predictive maintenance have demonstrable accuracy in retrospective studies. Yet the gap between a validated notebook model and an operational system that receives live data, updates automatically, and provides auditable decisions remains substantial in the upstream oil & gas industry. This research area addresses the engineering and governance challenges of that transition, drawing on the broader MLOps and software-engineering bodies of knowledge and adapting them to the constraints of OSDU-aligned upstream data platforms, SCADA latency requirements, and HSE-critical operational contexts.

Four interrelated sub-problems are addressed:

  • Continuous training and deployment: automating the pipeline from raw SCADA ingest through feature engineering to model retraining and blue–green deployment.
  • Model governance: versioned experiments, reproducible training environments, acceptance-test gates, and rollback capabilities.
  • Monitoring and drift detection: statistical tests on input distributions and prediction residuals to trigger retraining without human intervention.
  • Agentic orchestration: LLM-based agents that route queries across specialised sub-models, retrieve well-history context, and surface decisions with SHAP-based explanations.

3. Drift Detection and Retraining Triggers

Input drift for feature vector x is monitored using the Kullback–Leibler divergence between reference distribution Pref (training set) and current window Pt:

DKL(Pt ∥ Pref) = ∑j Pt,j log (Pt,j / Pref,j) (1)

The Population Stability Index (PSI), widely used in financial model monitoring and increasingly adopted in industrial analytics, aggregates drift across bins:

PSI = ∑b (Ab − Eb) ln(Ab / Eb) (2)

where Ab and Eb are actual and expected fractions in bin b. PSI < 0.1 indicates stability; PSI > 0.25 triggers retraining. Prediction-residual monitoring uses a CUSUM statistic with control limit h:

Ct+ = max(0, Ct−1+ + (et − k)),   alarm if Ct+ > h (3)

5. OSDU Data Platform Integration

The OSDU Data Platform [6] standardises upstream data schemas and APIs, enabling ML pipelines to consume well, production, and seismic data through a common interface. Key integration points include:

  • Schema registry: OSDU schemas for well logs (LAS3), production volumes (PRODML), and reservoir models (RESQML) drive automatic feature-store population without bespoke ETL.
  • Entitlements & governance: Data-access controls propagate to model training environments; sensitive production data remains within operator cloud tenancy.
  • Provenance lineage: Every derived feature links back to its source record ID, calibration certificate, and transformation code — satisfying audit requirements for HSE-critical applications.

6. Validity Threats and Open Problems

  • Train–serve skew: Differences between offline feature computation and online serving logic (time zones, aggregation windows) silently degrade model performance; point-in-time-correct feature stores are essential but add architectural complexity.
  • Data quality propagation: Upstream sensor faults produce corrupted features; robust serving pipelines must handle missing channels, clipped values, and physical-bound violations at inference time.
  • Model governance for safety-critical systems: Production systems affecting well control or injection pressure require a validation standard analogous to IEC 61508 functional safety; current MLOps practice lacks standardised acceptance criteria for this domain.
  • LLM hallucination in agentic systems: LLM-based orchestration agents may generate plausible but incorrect engineering recommendations; retrieval-augmented generation (RAG) with engineering knowledge bases and deterministic tool calls reduce but do not eliminate this risk.
  • Cost at scale: Real-time inference for hundreds of ESP wells at 1-minute cadence requires careful batching and model compression (quantisation, distillation) to remain economically viable on managed cloud endpoints.
