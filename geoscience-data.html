Geoscience & Upstream Data: Structures, Standards, and Methodology

This page consolidates the data taxonomies, storage formats, open standards, and preparation protocols that underpin data-driven research in subsurface energy systems. It is intended as a reference complementing the Research Topics pages, where these data types are deployed in specific studies.

1. Data Classification Across the Upstream Value Chain

Upstream data can be organised along three orthogonal axes: discipline (geophysics, geology, petrophysics, drilling, production, reservoir), temporal behaviour (static, depth-indexed, time-series), and acquisition channel (wireline, surface sensors, production telemetry, simulation). The classification used across this research programme is summarised in Table 1.

Class Temporal behaviour Representative variables Primary use
A — Static Time-invariant Tubing size, perforation depth, pump type & characteristic curve, wellbore trajectory Context for interpretation of dynamic data
B — Depth-indexed Function of depth Porosity, permeability, saturation, lithology, well logs, PVT curves, capillary pressure Reservoir characterisation, geological modelling
C — Time-series Function of time Rates, pressures, temperatures, motor current, vibration, choke position Surveillance, predictive analytics, control
D — Event Discrete events Workovers, failures, shut-ins, well tests, HSE incidents Labels for supervised learning, reliability analysis
E — Simulated / derived Model output Simulated pressure/saturation fields, history-matched parameters Surrogate modelling, design of experiments

Table 1. Five-class taxonomy used in subsurface data-driven studies.

5. Data Acquisition, Storage, and Lifecycle

A typical end-to-end upstream data pipeline moves measurements through five stages:

  1. Acquisition — downhole gauges, surface SCADA/DCS, fibre-optic DAS/DTS, rig-floor sensors.
  2. Edge / ingestion — MQTT, OPC-UA, Kafka, or WITSML-ETP brokers delivering telemetry with sub-second latency.
  3. Historisation — time-series databases (PI, IP.21, InfluxDB, TimescaleDB) with downsampling and compression policies.
  4. Enrichment — join with static master data (PPDM / OSDU), addition of engineering calculations, quality flags.
  5. Analytics & governance — feature stores, model registries, lineage tracking, access control.

6. Data Quality and Preparation Methodology

Downstream analytics are only as reliable as the preparation pipeline preceding them. The preparation protocol followed in this research programme comprises the following stages, adapted from Kirk [1] and Kelleher & Tierney [2]:

Standardisation:  x′ = (x − μx) / σx (1)
IQR outlier bounds:  [Q1 − 1.5·IQR,  Q3 + 1.5·IQR],  IQR = Q3 − Q1 (2)
Resampling (moving median):  x̃t = median{xτ : τ ∈ [t − Δ, t]} (3)
  1. Alignment & resampling: heterogeneous-frequency series are reduced to a common cadence using moving median or time-weighted averaging.
  2. Outlier treatment: IQR rule (Eq. 2), Hampel filter, or isolation forest; engineering limits preserve physically meaningful extremes.
  3. Gap handling: forward-fill for instrument stalls, linear / spline interpolation for short gaps, MICE for multivariate gaps.
  4. Normalisation / standardisation: Eq. 1 for distance-based models; min–max for bounded inputs.
  5. Feature engineering: lag features, rolling statistics, physical derived quantities (e.g. delta-pressure, fluid velocity).
  6. Split protocol: chronological train/validation/test splits for time-series — random splits induce optimistic bias.

8. Governance, Security, and Ethics

  • Confidentiality tiers: commercial production data is typically classified; anonymisation and aggregation rules must be defined before sharing.
  • Provenance & lineage: every derived feature should trace back to its raw source, tool calibration, and transformation code (reproducibility requirement).
  • Model governance: registered models, versioned datasets, and documented acceptance criteria are prerequisites for operational deployment.
  • Ethical use: environmental monitoring and HSE-relevant analytics require auditability comparable to safety-critical systems.
